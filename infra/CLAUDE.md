# mbgc-infra

Terraform source of truth for all mbgc cloud infrastructure.
One change here = one PR = one audit trail.

## Stack

- **IaC:** Terraform >= 1.14
- **State backend:** Supabase Storage (S3-compatible, bucket `mbgc-tfstate`)
- **Providers:** `hashicorp/google ~> 6.0` · `cloudflare/cloudflare ~> 5.0` · `supabase/supabase ~> 1.5`

## Layout

```
environments/prod/      # root module — prod is the only env
  main.tf               # resource declarations
  variables.tf
  versions.tf           # provider pins + S3 backend config
  providers.tf
  outputs.tf
  terraform.tfvars      # gitignored — written by bootstrap.sh
  backend.hcl           # gitignored — written by bootstrap.sh
modules/
  cloud-run/            # reusable: google_cloud_run_v2_service + IAM
  cloudflare-pages/     # reusable: cloudflare_pages_project
scripts/
  bootstrap.sh          # idempotent setup: GCP SA (no long-lived key), local files, GitHub secrets, terraform init
  smoke.sh              # post-apply smoke tests (Cloud Run services, DNS, gateway HTTPS)
.tflint.hcl             # tflint config (terraform + google rulesets)
.github/workflows/
  terraform.yml         # fmt → lint (tflint) → security (tfsec) → plan on PR / apply + smoke on main
  drift.yml             # weekly drift detection → GitHub issue
```

## Managed resources

| Provider | Resources |
|---|---|
| GCP Cloud Run | `myboardgamecollection` (monolith, Phase 5 decommission), `mbgc-gateway`, `mbgc-game-service`, `mbgc-importer-service`, `mbgc-auth-service` |
| Cloudflare | Pages project, DNS records for `lumedina.dev` (apex, www, api) |
| Supabase | Auth settings (`supabase_settings`) — JWT expiry, site URL, redirect URIs, session policy |

**Not managed by Terraform:**
- Cloud Run image, env vars, resources — owned by each service repo's CI/CD (`gcloud run deploy`).
- Cloudflare Pages config (`source` block, build settings) — configured via the CF dashboard. The resource uses `lifecycle { ignore_changes = all }` because the provider v5 sends malformed PATCHes for computed source fields.

## Secrets handling

- **Local:** `terraform.tfvars` and `backend.hcl` (both gitignored, generated by `bootstrap.sh`). GCP auth via ADC (`gcloud auth application-default login`).
- **CI:** `TF_VAR_*` injected from GitHub secrets; `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` for the S3 backend; Workload Identity Federation via `google-github-actions/auth@v2` for GCP (no long-lived key — uses `GCP_WORKLOAD_IDENTITY_PROVIDER` + `GCP_TERRAFORM_SERVICE_ACCOUNT`).
- **Service secrets** (`JWT_SECRET`, DB URLs, etc.) are set via `gcloud run services update --set-env-vars` or each service repo's CI/CD — never in this repo.

## Workflow

```sh
cd environments/prod
export AWS_ACCESS_KEY_ID=<supabase-s3-key>
export AWS_SECRET_ACCESS_KEY=<supabase-s3-secret>

terraform plan    # always review first
terraform apply
```

Never run `terraform apply` without reviewing the plan.

## Inspecting state on Supabase (S3 backend)

A named AWS CLI profile `supabase` is configured locally with the Supabase S3 endpoint:

```sh
aws --profile supabase s3 ls s3://mbgc-tfstate/prod/
aws --profile supabase s3api head-object --bucket mbgc-tfstate --key prod/terraform.tfstate
aws --profile supabase s3 cp s3://mbgc-tfstate/prod/terraform.tfstate -   # stream to stdout
```

`aws s3 cp` requires both source and destination — use `-` for stdout, or a local path.
Curl with `-u` will not work against Supabase S3: it requires SigV4 signing.

## Bootstrap (first run or credential rotation)

```sh
sh scripts/bootstrap.sh
```

Idempotent — safe to re-run. Uses env vars if already set, prompts only for missing values.
Writes `backend.hcl` + `terraform.tfvars`, syncs all GitHub secrets, runs `terraform init`.

Bootstrap must be run **twice** on first setup — once before `terraform apply` (to create the GCP SA and write credentials), and once after (to read WIF outputs from state and push `GCP_WORKLOAD_IDENTITY_PROVIDER` + `GCP_TERRAFORM_SERVICE_ACCOUNT` to GitHub).

IAM roles granted to the terraform SA (required for CI):
`run.admin`, `iam.serviceAccountUser`, `iam.serviceAccountAdmin`, `iam.workloadIdentityPoolAdmin`, `artifactregistry.admin`, `resourcemanager.projectIamAdmin`, `serviceusage.serviceUsageAdmin`

## CI

- **`terraform.yml`** — triggered on PR / push to `main` for `environments/**` and `modules/**`.
  Pipeline: `fmt` + `tflint` + `tfsec` (all must pass) → `plan` (with `-detailed-exitcode`, posts single updating PR comment) → `apply` on merge to `main` → `smoke.sh`.
- **`drift.yml`** — runs Monday 09:00 UTC. Opens/updates a `drift`-labeled GitHub issue if plan shows changes.

## Tests

- **`scripts/smoke.sh`** — verifies expected Cloud Run services exist in `us-central1`, `api.lumedina.dev` resolves, and the gateway returns non-5xx over HTTPS. Runs post-apply in CI; safe to run locally (`sh scripts/smoke.sh`). Override target with `GCP_PROJECT`, `GCP_REGION`, `API_HOST` env vars.
- **`tflint`** — uses the terraform preset + google ruleset, configured in `.tflint.hcl`.
- **`tfsec`** — static analysis for GCP/Cloudflare misconfigurations. Public-invoker IAM on the monolith/gateway is intentional; silence false positives with inline `# tfsec:ignore:<rule-id>` rather than disabling the check globally.

## Provider limitations

- `hashicorp/google ~> 6.0` — official, current. Cloud Run image/env/resources are ignored via `lifecycle.ignore_changes` because deployments are owned by each service repo's CI/CD. **Both** scaling blocks are ignored:
  - `template[0].scaling` (per-revision, min/max instances)
  - top-level `scaling` (service-level manual scaling — `manual_instance_count`, `scaling_mode`; added in provider 6.50.0)
  GCP returns computed defaults for both (`manual_instance_count = 0`, etc.), so omitting either causes perpetual no-op drift on every plan.
- `cloudflare/cloudflare ~> 5.0` — official, current. Pages `source` block (GitHub integration) is ignored via `lifecycle.ignore_changes` because it is managed through the Cloudflare dashboard.
